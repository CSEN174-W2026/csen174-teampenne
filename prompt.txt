Optimize: average response time across jobs.
Needs (values):

Helpful policies:
Least-loaded: usually reduces average latency under uneven load.


Latency-aware (if you add it later): choose node with best predicted completion time.

2) Reduce Tail Latency (p95 / p99)

Files: agentic_controller.py
This file is the brain of your system. It represents the control plane that observes the state of all nodes and makes decisions about where jobs should go. Conceptually, this is your ‚Äúagent‚Äù ‚Äî it gathers telemetry (CPU, memory, queue length, latency, throughput, etc.) from each node and then applies scheduling logic (your policies and goals) to choose actions. It sits above the nodes and does not execute jobs itself; instead, it reasons about the cluster and tells nodes what to do.
In terms of flow, the controller typically:
Discovers nodes (often via virtualbox.py)
Pulls current metrics using NodeClient
Applies policies from policies.py
Chooses a node
Submits jobs to that node
This is where your ‚Äúagentic‚Äù aspect lives ‚Äî multi-metric reasoning, tradeoffs, and future learning logic all belong here.

node_worker.py
This file implements the node-side server. Each VM runs one copy of this, and it simulates (or executes) work. It exposes a FastAPI service with endpoints like /metrics and /submit. When the controller sends a job, this file is responsible for queueing it, running it (currently simulated via asyncio.sleep()), and updating internal state such as in_flight, queue length, latency history, and throughput.
Even though some parts are simulated, this file is modeling the runtime behavior of a compute node. It blends real OS telemetry (via psutil) with system-level state (job queue, EWMA latency, job completion history). This separation makes the system realistic while still controllable for experimentation.

node_client.py
This file is the network interface between the controller and nodes. It wraps HTTP requests so the controller doesn‚Äôt have to deal with raw REST calls. Instead of writing requests.get(...) everywhere, the controller just calls client.get_metrics(node) or client.submit_job(node, job).
This abstraction is important because it isolates communication logic from scheduling logic. If you later changed from HTTP to gRPC, or added retries, authentication, or logging, you‚Äôd update this file without touching the controller logic.

state_types.py
This file defines the shared data structures used across the entire system. It contains dataclasses like NodeSnapshot and JobRequest, which represent the state of a node and the structure of a job. These act as the language spoken between the controller, client, and nodes.
The purpose of centralizing these definitions is consistency. Instead of passing around loose dictionaries, the whole system uses structured objects. This reduces bugs, makes code more readable, and clearly defines what ‚Äúnode state‚Äù and ‚Äújob‚Äù mean in your architecture.

Policies List: Tier 0 ‚Äî Easiest (no VM stats required)

1) Random
What it actually does:
 For every job, pick a node uniformly at random. No memory, no state, no load awareness.
VM data needed: none
 Best goals: none (baseline), sometimes helps Fairness by chance
 Hurts: Mean latency, Tail latency, SLA, Load balance
Role: exploration baseline and comparison point.

2) Round Robin
What it actually does:
 Maintain an index. Send job 1 ‚Üí node A, job 2 ‚Üí node B, job 3 ‚Üí node C, then wrap around. Ignores node state.
VM data needed: none
 Best goals: Fairness, Stability/variance, Load balance (if nodes equal)
 Hurts: Mean/Tail/SLA under bursts or node heterogeneity
Conflict: fairness vs tail latency.

3) Weighted Round Robin
What it actually does:
 Like round robin, but some nodes get more turns based on predefined weights.
 Example: A, A, B, C if weights = [2,1,1].
VM data needed: none (weights are config)
 Best goals: Throughput (favor faster nodes), Fairness (if weights per class), Load balance
 Hurts: Tail/SLA if workload shifts or weights mismatch reality

Tier 1 ‚Äî Easy (needs per-node load number)

4) Least-Loaded
What it actually does:
 Query each node‚Äôs queue length or in-flight job count and pick the smallest.
VM data needed: queue_length or in_flight_jobs
 Best goals: Mean latency, Tail latency, SLA, Load balance, often Throughput
 Hurts: can reduce Fairness

5) Power-of-Two Choices
What it actually does:
 Pick 2 random nodes, compare their loads, choose the less loaded of the two.
VM data needed: same as least-loaded
 Best goals: Mean latency, Tail latency, Load balance
 Why it‚Äôs great: nearly as good as least-loaded with less coordination.

6) Load Threshold (JSQ with cap)
What it actually does:
 Pick least-loaded, but if its queue exceeds a threshold, skip it and try the next best node.
VM data needed: queue_length
 Best goals: SLA violations, Tail latency, Anti-starvation
 Hurts: sometimes Throughput

Tier 2 ‚Äî Moderate (needs job size or node speed)

7) Fastest-Node Bias
What it actually does:
 Prefer nodes known to be faster (static weights or periodic benchmarks). Sends more jobs to those nodes.
VM data needed: none if preconfigured; optional benchmarking
 Best goals: Throughput, Flow time
 Hurts: Load balance, Tail/SLA

8) Size-Aware Routing (SJF-style)
What it actually does:
 If job is small ‚Üí send to lightly loaded node;
 If job is large ‚Üí avoid congested nodes or handle differently.
VM data needed: job size estimate
 Best goals: Flow time, Mean latency
 Hurts: Fairness, Anti-starvation

9) Min Estimated Completion Time (MECT)
What it actually does:
 For each node, estimate:
completion_time = current_wait + job_size / node_speed

Choose the smallest.
VM data needed: queue length + node speed estimate + job size
 Best goals: Mean latency, Flow time, often Throughput
 Hurts: Fairness, sometimes Load balance

Tier 3 ‚Äî Harder (history, learning, class-aware)

10) Latency-Aware (EWMA)
What it actually does:
 Keep a moving average of observed latency per node. Prefer nodes that recently performed well.
VM data needed: none (agent measures latency)
 Best goals: Mean latency, Tail latency, SLA
 Hurts: Load balance, Stability

11) Tail-Guard / Straggler-Aware
What it actually does:
 Track which nodes recently caused slow jobs or SLA misses. Penalize those nodes temporarily.
VM data needed: agent-side history
 Best goals: Tail latency, SLA, Stability
 Hurts: Throughput

12) Fairness-Aware Scheduler
What it actually does:
 Track per-user performance. Prefer decisions that improve the worst-off user or keep users balanced.
VM data needed: job user/class label
 Best goals: Fairness, Anti-starvation, Stability
 Hurts: Throughput, sometimes Mean/Tail latency
